# DOS Java プロジェクト計画サマリー

## プロジェクト概要

16bit PC-DOS 上で動作する軽量 Java サブセット実装を、Open Watcom v2 C コンパイラでクロスコンパイル可能なソースコードとして作成します。

## 主要な設計決定

### 1. ターゲット環境
- **プラットフォーム**: 16bit PC-DOS (MS-DOS 互換)
- **メモリモデル**: Small (64KB コードセグメント + 64KB データセグメント)
- **動作モード**: リアルモード (640KB メモリ制限)
- **対象 CPU**: 8086/8088 以上

### 2. Java 実装レベル
- **独自の軽量 Java サブセット**を実装
- 16bit 環境に最適化された最小限の言語機能
- 基本的な java.lang パッケージのみ

### 3. バイトコード処理方式
- **独自の軽量バイトコード形式 (.djc)** を使用
- 標準 .class ファイルから .djc 形式に変換するプリプロセッサを作成
- 16bit 環境に最適化された命令セット

## システムアーキテクチャ

```
Java ソース (.java)
    ↓ javac
Java バイトコード (.class)
    ↓ djpreproc (プリプロセッサ)
DOS Java バイトコード (.djc)
    ↓ dosjava.exe (VM)
実行結果
```

### コンポーネント構成

1. **DOS Java 仮想マシン (dosjava.exe)**
   - クラスローダー
   - バイトコードインタープリタ
   - メモリマネージャ
   - ガベージコレクタ (参照カウント方式)
   - スタック管理

2. **プリプロセッサ (djpreproc.exe)**
   - .class ファイルパーサー
   - バイトコード変換器
   - .djc ファイルライター

3. **ランタイムライブラリ (C 実装)**
   - java.lang.Object
   - java.lang.String
   - java.lang.System
   - java.lang.Integer
   - java.lang.Boolean

## サポートする Java 機能

### データ型
- **プリミティブ**: `int` (16bit), `boolean`, `char`
- **参照型**: `Object`, `String`, 配列 (制限付き)

### 言語構造
- クラス (単一継承のみ)
- メソッド (インスタンスメソッド、静的メソッド)
- フィールド (インスタンスフィールド、静的フィールド)
- 制御構造: if/else, while, for
- 演算子: 算術、比較、論理

### サポートしない機能
- long, float, double 型
- インターフェース
- 抽象クラス
- 例外処理 (try/catch)
- マルチスレッド
- リフレクション
- 内部クラス
- ジェネリクス
- アノテーション

## 軽量バイトコード形式 (.djc)

### ファイル構造
```
┌─────────────────┐
│ ヘッダー         │ - マジックナンバー (0x444A)
│                 │ - バージョン
│                 │ - 定数プール数
│                 │ - メソッド数
├─────────────────┤
│ 定数プール       │ - 文字列定数
│                 │ - 整数定数
│                 │ - クラス参照
├─────────────────┤
│ メソッド記述子   │ - メソッド名
│                 │ - 引数/戻り値型
│                 │ - コードオフセット
├─────────────────┤
│ フィールド記述子 │ - フィールド名
│                 │ - 型
│                 │ - フラグ
├─────────────────┤
│ バイトコード     │ - 最適化された命令列
└─────────────────┘
```

### 主要な命令
- `PUSH_CONST` - 定数をスタックにプッシュ
- `ADD`, `SUB`, `MUL`, `DIV` - 算術演算
- `LOAD_LOCAL`, `STORE_LOCAL` - ローカル変数アクセス
- `GOTO`, `IF_TRUE`, `IF_FALSE` - 制御フロー
- `INVOKE_STATIC`, `INVOKE_VIRTUAL` - メソッド呼び出し
- `NEW` - オブジェクト生成
- `GET_FIELD`, `PUT_FIELD` - フィールドアクセス
- `RETURN` - メソッドからの復帰

## メモリレイアウト (Small モデル)

```
データセグメント (64KB):
┌──────────────────┐ 0xFFFF
│ スタック (↓)     │ ~20KB
├──────────────────┤
│ ヒープ (↑)       │ ~10-14KB (オブジェクト用)
├──────────────────┤
│ 静的データ       │ ~10KB (VM 状態、クラスデータ)
├──────────────────┤
│ BSS              │
├──────────────────┤
│ 初期化済みデータ │
└──────────────────┘ 0x0000

コードセグメント (別の 64KB):
┌──────────────────┐
│ VM コード        │ ~40KB
│ ネイティブメソッド│ ~15KB
│ (予備)           │ ~9KB
└──────────────────┘
```

## 技術的制約と対策

### 制約
1. **メモリ制限**: 64KB データセグメント
   - **対策**: 効率的なデータ構造、参照カウント GC

2. **パフォーマンス**: インタープリタのオーバーヘッド
   - **対策**: ホットパスの最適化、命令のインライン化

3. **機能制限**: 浮動小数点演算なし
   - **対策**: 整数演算のみに限定、ドキュメントで明示

4. **デバッグ困難**: DOS 環境での開発
   - **対策**: 詳細なログ機能、DOSBox でのテスト

## リスク管理

### 技術的リスク
- **メモリ制約が厳しすぎる**: Compact モデルへの移行を検討
- **パフォーマンスが遅すぎる**: JIT コンパイルの検討
- **Open Watcom 互換性問題**: 標準 C 機能のみ使用

### スケジュールリスク
- **複雑さの過小評価**: MVP を優先、段階的に機能追加
- **デバッグの困難**: 早期にデバッグツールを構築
